## Decision #79 - 2026-05-11
**Context:** Country attribution on CCN/CRN nodes used to live inside the `if (layers.has("geo"))` block of `buildGraph`, so `n.country` was only populated when the geo layer was on. After adding a "Location" row to the CCN/CRN detail panels (showing `🇫🇷 France` etc.), the row was missing whenever Geo was off (the default).
**Decision:** Split the geo block into two passes in `network-graph-model.ts`: (1) country *attribution* (set `n.country` on located CCN/CRN, build the `represented` set) always runs; (2) geo *layer artifacts* (the country hub nodes + geo edges) only run when `layers.has("geo")`. Detail panels read `node.country` regardless of layer state.
**Rationale:** Country is a property of the node, not of a presentation layer. Coupling it to the layer toggle was a quirk of the original geo-layer commit (#74), not a design intent. The split is mechanical, doesn't change geo-layer visuals, and unlocks the Location row without forcing users to enable Geo to see where a node lives. The cost is one extra pass over `nodes` when Geo is off — negligible at ~500 nodes.
**Alternatives considered:** Look up country directly in the panel from `node-locations.json` (rejected — duplicates the existing graph-model lookup; two sources of truth for the same data). Force the geo layer on by default (rejected — the layer affects the entire layout; turning it on just to show a country label is overkill). Add a separate `geo-attribute` non-visible layer that only sets `country` (rejected — adds a layer concept the user can never toggle, confusing).

## Decision #78 - 2026-05-11
**Context:** Address deep-link (`?address=0x...`) previously highlighted matching nodes with a pulsing primary-color ring and auto-fit the camera to them, but otherwise nothing changed — no detail panel, no spotlight dim. After making `?selected=` open a node panel, search-by-address still felt like nothing happened: pulse rings were easy to miss against a busy graph and the user got no panel feedback. Separately, `highlightedIds` was computed only from `owner === address`, so a wallet that staked (id-match) or received rewards (reward-match) but didn't own any node showed zero highlights.
**Decision:** Treat `?address=` as a first-class selection signal. (1) Open a dedicated `NetworkSearchAddressPanel` (right side, 280px, same chrome as the node panel) when address is set and no node is selected — showing the copyable address, link count, an "Open wallet view →" link, and a staking section listing every CCN where the address appears in `c.stakers` with per-position and total ALEPH. (2) Expand `highlightedIds` to match `id` (staker), `owner`, or `reward` so the on-graph pulse reflects the wallet's full footprint. (3) Extend the selection spotlight: when no node is selected but `highlightedIds.size > 0`, `relevantIds = highlightedIds ∪ 1-hop neighbors` and the existing dim path applies to nodes, labels, *and* edges (non-incident edges fade). Address-panel close clears `?address=` *and* the search input, matching the panel-close clears-search pattern.
**Rationale:** The pulse-ring-only design assumed power users would notice the dim signal — in practice users searched an address and saw "nothing happen." Promoting the address to a real panel + spotlight reuses the same visual idiom users already know from node selection (dim everything else, panel on the right), so address search becomes discoverable without inventing a new pattern. Expanding `highlightedIds` to id/owner/reward matches the wallet's actual footprint, not just its ownership — a staker who doesn't own anything still gets visual presence. The edge-dim extension was a missing piece: previously only `selectedId` drove the `faded` flag, so address search left edges fully lit while nodes dimmed — visually inconsistent.
**Alternatives considered:** Auto-select the first matching node instead of opening an address panel (rejected — the panel context would be "this node" rather than "this wallet"; misleading when the address owns multiple nodes). Treat the address as a virtual GraphNode and add `kind: "address"` (rejected — ripples through `RADIUS`, color tables, panel switch statements for a UI-only synthetic). Use a thicker pulse ring instead of dimming (rejected — pulse alone didn't read; the dim is what makes the spotlight feel intentional). Keep `highlightedIds` owner-only (rejected — a real-world staker without ownership is invisible, contradicts the new panel's "Linked to N nodes" copy).

## Decision #77 - 2026-05-11
**Context:** Country nodes on `/network` rendered as flat black circles instead of cyan-teal. The CSS token `--network-country` declared in `globals.css` resolved to an empty string in the browser; SVG `fill="var(--network-country)"` then fell back to the SVG default (`black`). Sibling tokens (`--network-edge-owner`, `--network-edge-reward`) in the same `:root` block compiled fine. The compiled `_next/static/chunks/src_app_globals_*.css` literally did not contain `--network-country` — the property was being stripped at build time.
**Decision:** Drop the chroma value on `--network-country` from `0.16` (light) / `0.14` (dark) to `0.13` for both, keeping hue `200°` and lightness `0.70` / `0.78`. The token now compiles through Lightning CSS as a valid `lab()` value with a hex fallback.
**Rationale:** Lightning CSS (Tailwind v4's CSS engine) silently drops `oklch()` values that are out of sRGB gamut at the requested hue, instead of clamping. At hue ~200° (cyan), the in-gamut chroma ceiling is around `0.13` at these lightnesses — `0.16`/`0.14` overshot and got dropped silently. The other tokens used `230°` (blue), which has more headroom and stayed in gamut. The visual difference between chroma `0.13` and `0.14`/`0.16` is small enough to not warrant a different hue or a hex fallback, and keeping OKLCH preserves the codebase convention.
**Alternatives considered:** Switch the token to a hex value (rejected — breaks the OKLCH convention used by the rest of `globals.css`; would also lose automatic light/dark inversion since hex doesn't communicate intent). Move to a different hue with more headroom (rejected — the cyan/teal hue was the entire point of the choice in Decision #74; would lose the distinct identity vs CCN purple, CRN green, staker amber). Add a hex fallback alongside the OKLCH (rejected — works but doesn't address the root cause of *why* the OKLCH dropped, and would mask the issue if anyone later swapped values back to out-of-gamut). Verified with browser-resolved computed styles after each chroma value.

## Decision #76 - 2026-05-11
**Context:** Decision #74's initial design treated geo edges as force-only — `network-graph.tsx` early-returned on `e.type === "geo"` so they never rendered, on the theory that proximity to the country dot would already convey "in this country." Real-world testing with Geo + Structural both on showed the cross-country CCN→CRN structural arrows visually muddled the geo grouping — each country dot was surrounded by its cluster but no edges connected them, so the eye couldn't tell which CCN belonged to which country at a glance.
**Decision:** Render geo edges as a thin, country-tinted dashed tether — `STROKE.geo = var(--network-country)`, `DASH.geo = "1 2"`, `OPACITY.geo = 0.35`, `strokeWidth=0.5`, no arrowhead. The country becomes a visible hub-and-spoke hub. Wire country into the selection-incident-color path (`incidentColor` returns `var(--network-country)` when a country is selected) so picking a country brightens its tethers to `0.9` opacity via the existing `highlightColor` branch in `NetworkEdge`. Add a "Country tether" line swatch to the legend (also conditional on the geo layer).
**Rationale:** Cheapest, highest-signal fix for the stated complaint. The dash pattern (`1 2` — short dash, double gap) is distinct from owner's `1.5 1` and reward's `0 0.4` dots, so the four edge types each read as their own thing. Country color reinforces the grouping semantics — tethers and country dots share a hue so the eye groups them naturally. Lifting the gate is a two-line code change (the render early-return + the `OPACITY.geo` bump from `0` to `0.35`), and "always rendered when Geo is on" matches the mental model "layer = visible thing" — toggling Geo off hides both the country dots and their tethers in one motion. The selection-incident wiring is a single switch case addition; without it, selecting a country wouldn't brighten its tethers and the country's spotlight would feel weaker than CCN/CRN selection.
**Alternatives considered:** Keep force-only (rejected — original instinct, but the user feedback showed proximity alone wasn't enough when Structural is co-active). Render only when Geo + Structural both on (rejected — breaks "layer = visible thing", silently changes visuals when toggling Structural, and requires plumbing `layers` into `NetworkGraph` which currently doesn't have it). Solid neutral hairline like structural at low opacity (rejected — too similar to structural; the eye would conflate them). Render geo edges + dim structural when both on (rejected — demotes the most informative edges, bad tradeoff). Arrowheads on tethers (rejected — membership is non-directional; arrows would imply a wrong semantic).

## Decision #75 - 2026-05-11
**Context:** The `/network` focus-pill back-button (`‹`) called `router.back()` to step up one layer in the focus chain. CLAUDE.md's earlier design (#73) leaned on the browser history stack — each Focus action `router.push`es, selection clicks `router.replace`, so back walks the chain. Real-world use surfaced the flaw: if a user arrives at `/network` from a parent route (e.g., `/admin/network` in a host shell), the *first* Focus push sits on top of that history. Clicking the pill back-button rewinds past the first focus *and out of the dashboard entirely*, landing on `/admin/network`.
**Decision:** Encode the focus chain in the URL as a comma-separated `?focus=A,B,C` list (last = active). `useNetworkGraph` parses via a new `parseFocusStack()` and exposes both `focusId` (last) and `focusStack` (full array). Each Focus action appends to the stack and `router.push`es; selection clicks still `router.replace`. The pill back-button (`onStepBackFocus`) pops the last entry and `router.replace`s the trimmed URL — never calls `router.back()`. When the stack reaches empty, both `focus` and `selected` are dropped.
**Rationale:** The URL is now the source of truth for the focus chain, so the pill can never leave `/network` regardless of how the user arrived there. Bookmarkability is preserved — pasting a multi-element focus URL restores the chain. Browser back/forward still walks the same chain because each Focus is `router.push`, so power-users who use the browser controls aren't penalized. The behavior matches the documented mental model ("step up one focus layer") without depending on the browser-history side-effect that made it fragile.
**Alternatives considered:** Same-route guard before `router.back()` (rejected — there's no reliable way to peek at the history entry's URL before navigating). Push a sentinel history entry at page mount as a "back-stop" (rejected — pollutes history with no-op entries; user would see the page URL flicker on browser back). Track the focus stack in `useRef` / sessionStorage (rejected — refresh and cross-tab navigation lose the chain; URL-encoded is bookmarkable). Make the pill back-button just clear focus (rejected — collapses the back and × buttons into one, loses the chain-walk feature the user actually relies on for drilling country → CCN → CRN).

## Decision #74 - 2026-05-11
**Context:** The `/network` force-directed graph reads relational structure beautifully (CCN↔CRN parents, owner cliques, staker pools, reward clusters) but says nothing about how nodes group geographically. The worldmap card on the Overview hero already plots active nodes by country, and `src/data/node-locations.json` + `src/data/country-centroids.json` are already loaded and refreshed on every build. Backlog asked four questions: (a) optional layer toggle vs. dedicated mode, (b) how to handle nodes with no resolved location, (c) force simulation vs. fixed geo positions, (d) interaction with existing focus/spotlight features.
**Decision:** Add a fifth `geo` layer alongside structural/owner/staker/reward (off by default). When on, `buildGraph` emits a `kind: "country"` node per represented country (id namespace `country:<ISO>`) and one `type: "geo"` edge per located CCN/CRN to its country. **Layout is force-driven, not projected** — country nodes are regular sim nodes (no `fx`/`fy` pin); the force simulation places them by cluster mass. Geo edges run through a separate `forceLink` at `distance=25, strength=1` so the existing relational `forceLink` keeps d3's degree-aware default strength; country charge in `forceManyBody` is boosted to `-800` (vs `-180` for other nodes) so country clusters can't overlap. Geo edges render as country-tinted dashed tethers (see Decision #76 for the visibility iteration); country labels bypass `LABEL_ZOOM_THRESHOLD` so the grouping is always legible. Country is the top-tier visual when the layer is on (`RADIUS.country=22`, bigger than CCN's 16, with full node treatment — underlay/fill/stroke/outer ring/selection halo — and a new `--network-country` cyan/teal token). Countries are clickable, focusable, and searchable — a country match in the search bar fires `?focus + ?selected`. The detail panel adds a `NetworkDetailPanelCountry` body with flag emoji + CCN/CRN/owner stat tiles aggregated from incident geo edges. Stakers/rewards/unlocated nodes have no geo target and float as today. No world-map backdrop.
**Rationale:** A layer toggle (over a dedicated mode/route) keeps the mental model — `/network` is one relational graph; geography is one more way to read it. Force-driven country placement (over real-world pinning via `networkMercator`) emerged from real-world testing: pinning at projected centroids put neighbors (UK/NL/DE/PL/FR) too close together while leaving the rest of the canvas mostly empty, and combining Geo with Structural produced a tangled mess of cross-country arrows. Unpinning the country and letting the simulation place it by cluster mass naturally distributes countries by how much they have to anchor — heavy clusters (FR/US/DE) drift to the edges where they have room, smaller ones settle in between, and clusters can't overlap because country charge dominates. Making country a real node kind (rather than a label-only annotation) gives click/focus/search for free via the existing infrastructure — the ego-subgraph logic already walks edges, so focusing a country naturally pulls in its located nodes and *their* structural/owner/staker neighbors. Skipping the world-map backdrop avoids diluting the relational reading; the worldmap card on Overview already serves the geographic-glance use case.
**Alternatives considered:** Real-world projected centroids via a `networkMercator` Mercator (rejected after testing — central Europe overlapped, rest of canvas was dead space). Dedicated `/network/geo` mode (rejected — duplicates the layer machinery for no win). Hard-pin all located CCN/CRN at their country centroid (rejected — kills cross-country owner/staker readability). Country as a label-only annotation (rejected — loses click/focus/search; would need a parallel label-positioning layer). World-map SVG behind the graph (rejected — clutter; the worldmap card is the geography-first surface). City/region granularity in v1 (deferred — needs larger geo DB). Per-link strength override on a single `forceLink` (rejected — forces us to reimplement d3's degree-aware default strength for non-geo edges; cleaner to split into two `forceLink` instances). Per-kind charge via `forceManyBody().strength((d) => ...)` chosen over `forceCollide` for inter-country separation because charge interacts smoothly with the rest of the sim, while `forceCollide` introduces hard-radius constraints that look mechanical.

## Decision #73 - 2026-05-09
**Context:** Real-world use of the `/network` detail panel surfaced five problems: (a) the 400px right-edge slide-in covered most of the map; (b) it extended to `top: 0` and visually clashed with the search bar / layer toggles; (c) the Focus and Close buttons sat in their own header above `<NodeDetailPanel>` so they read as separate from the node card; (d) CCN nodes always showed empty content because `useNode(hash)` is the scheduler API and CCNs aren't in that dataset; (e) CRN content was a copy of the full `/nodes?view=…` detail card — most of it (GPU lists, VM lists, history table, CPU features) wasn't graph-relevant.
**Decision:** Replace the slide-in with a 280px floating card (`right-4 top-20 bottom-4`, `bg-background` with `rounded-xl border shadow-md`). Split the panel into a shared shell (`network-detail-panel.tsx`) plus three presentational bodies (`-ccn`, `-crn`, `-address`) selected by `node.kind`. Expose `nodeState` from `useNetworkGraph` so the shell can resolve `CCNInfo`/`CRNInfo` directly — the CCN body and the parent-CCN lookup for CRNs need no extra API calls; only the CRN body keeps `useNode(hash)` for resource bars and VM count. CCN body shows score, CRNs/Stakers stat tiles, total staked, owner, reward; CRN body shows status, VM count, parent-CCN clickable, CPU/Memory bars, owner; staker/reward body shows the address with copy + wallet link plus a degree summary against the visible graph. Heavy content (GPU/VM/history) drops in favor of a single "View full details →" footer link to `/nodes?view=<hash>`. The Focus action sets both `?focus` and `?selected` so the panel stays open after focusing.
**Rationale:** Each complaint was structural. Size and position weren't tunable in isolation — the panel needed to coexist with the toolbar, the focus banner, and the map at the same time, which a 400px slide-in over `top: 0` couldn't do. The empty CCN cards were a data-source bug: the panel called `useNode` (scheduler-only) for any CCN/CRN, which silently returned `undefined` for CCNs. Rather than add a CCN-specific fetcher, exposing the already-loaded `nodeState` from `useNetworkGraph` is zero new requests and keeps each body trivially testable. Trimming the CRN body to graph-relevant facts is YAGNI applied to a panel context — when the user wants the full report they have a one-click route to `/nodes?view=`. Setting both `?focus` and `?selected` on the Focus action turns a previously disruptive operation (panel closes, user clicks the focused node again) into a single click. Putting Focus and × inside the card header solves the disconnection complaint without any new behavior — they were already attached, just visually orphaned by the wrapper layout.
**Alternatives considered:** A floating tooltip-style card next to the selected node (option A in brainstorming, rejected — too small for the "Rich" content the user wanted). A bottom dock (option C, rejected — wide and shallow doesn't fit lists of stakers / parent CCN well). Floating card with "open full details →" replacing all body content (option D, rejected — user wanted "a bit more information than that" inline). Loading CCN content via a new dedicated fetch (rejected — `useNetworkGraph` already has the data; an extra hook would be redundant). Animating the panel position between selections (rejected — adds complexity for marginal value; a simple unmount + remount is enough).

## Decision #72 - 2026-05-09
**Context:** Real-world use of `/network` after #70/#71 surfaced four UX gaps: (a) on reset-view (and the first load), the layout visibly expanded outward from a tight Fibonacci spiral for several seconds before the camera re-fit — felt like a multi-second swoop where the first ~9s nothing was settled and the camera then "snapped" into place; (b) at default zoom (k≈0.3 once the full graph fits) nodes were nearly invisible — a `r=13` CCN at `k=0.3` is a 3.9px screen blob with an `0.18` translucent fill and a `0.225px` stroke; (c) edges were dim — structural at `0.4` opacity faded into the dark background, and the `owner` overlay's saturated blue dashes at `0.25` competed with the structural backbone for visual priority despite carrying secondary information; (d) selecting a node coloured its incident edges and added a halo, but unrelated nodes/edges stayed at full strength and visually drowned the spotlight.
**Decision:** (1) **Pre-warm the simulation synchronously on fresh mounts.** The `simNodes` useMemo detects an empty `positionsRef` (initial load or reset-view's `key` bump in `network/page.tsx`), constructs a throwaway `forceSimulation` with the same charge/link forces, runs `tick(300)` synchronously, writes the converged positions back to `positionsRef`, and signals via a `justWarmedUpRef` ref. The live simulation effect then creates its real sim with `alpha(0)` so it doesn't re-shake the converged layout. The page-level `SETTLE_MS` drops `2200 → 500` to match the actual settle time (just the 450ms refit transition). (2) **Bigger node radii + zoom-adaptive size.** Bases bumped: CCN `13→16`, CRN `8→11`, staker `3→5`, reward `4→6`. A `nodeScaleForZoom(k)` helper returns a multiplier that's `1` in `[0.6, 1.5]`, boosts up to `~1.9×` at low zoom, eases to `0.7×` at high zoom, quantized to `0.1` steps; threaded through as a `sizeScale` prop on `NetworkNode` and applied to all radii (fill, halo, CCN outer ring, reward rect) plus the structural arrow size and label gap. (3) **Edge brightness pass.** Structural opacity `0.4 → 0.6`. Owner overlay color reset from `var(--network-edge-owner)` (saturated blue) at `0.25` to `currentColor` (the same neutral gray as structural) at `0.2`, keeping the `1.5 1` dashed pattern and `0.5px` stroke. Legend swatch updated to match. **Arrowheads on structural CCN→CRN edges** via a single `<marker id="arrow-end">` defined in the SVG `<defs>`, sized `10 × nodeScale` user-space units, with `fill="context-stroke"` so the arrow inherits whatever color the line has (gray default, kind-color highlight, dim faded). The line's target endpoint is shortened by `RADIUS.crn × nodeScale + 1.5` so the arrow tip lands just outside the CRN border — without that backoff the marker would render at the line endpoint (CRN center) and be masked by the node's opaque background underlay. Implemented in `network-graph.tsx` (per-edge math + `<defs>` + `withArrow` boolean) and a `withArrow?: boolean` prop on `NetworkEdge` that toggles `markerEnd`. (4) **Selection spotlight.** When `selectedId != null`, a `relevantIds = {selected} ∪ direct-neighbors-via-any-currently-visible-edge` set is built once per `(selectedId, graph)` change. `NetworkNode` gains a `dimmed` prop that drops group opacity to `0.18` (precedence above the existing `inactive=0.6` so unrelated nodes recede uniformly); the existing `faded` prop on `NetworkEdge` is now wired to `selectedId != null && !isIncident`; the floating label layer also dims to `0.18` for non-relevant nodes.
**Rationale:** The spiral-then-expand-then-refit sequence was a layered consequence: reset bumped a `key` on `<NetworkGraph>` which discarded `positionsRef` and re-seeded the Fibonacci spiral, the initial RAF refit fitted that *spiral* (zooming in tight), the live sim then ran ~135 ticks at `alphaDecay=0.05` to spread the layout out (during which nodes drifted off-screen since the camera was still on the spiral), and only on `sim.on("end")` did refit fire again to fit the converged graph. Each piece was correct in isolation; together they made reset feel broken. Pre-warming inside the useMemo (in render phase, not as an effect) means the converged positions are written into `positionsRef` *before* the first paint, so the first React commit and the initial RAF refit both see the spread layout. Choosing `tick(300)` over `(135)` gives slack for slower convergence at higher node counts; the work is bounded (Barnes–Hut `O(N log N)` × 300 is well under 100ms for ~500 nodes) and runs once per fresh mount, not per data refetch (`positionsRef.current.size === 0` gates it). The live sim's `alpha(0)` start matters because re-creating the simulation with default `alpha=1` on already-converged positions would re-apply forces with full strength and re-shake everything for ~135 ticks even though the user perceives no benefit. Bigger radii alone wouldn't have been enough — at low zoom the screen size is `world_radius × k`, so we needed the radius to *grow* as `k` shrinks; the partial inverse `1/k` clamp gives nodes a near-constant screen size in the central zoom band while still letting them grow modestly at extreme low zoom and shrink at extreme high zoom. Quantization to `0.1` is the perf hedge: without it a smooth wheel-zoom event stream would re-render 500+ memoized nodes on every event. Owner edges going neutral instead of blue is a deliberate de-prioritization — the structural backbone IS the network's primary topology, and the overlay layers should read as annotation patterns, not parallel signals; using the same hue keeps the eye on the structural shape and the dashed pattern alone tells you it's owner. Arrowheads using `context-stroke` (SVG2, fully supported in modern Chromium/Firefox/Safari since 2022–23) avoids a per-color marker variant and means the highlight color when an edge is incident to a selected node carries through to its arrow without separate state. Selection spotlight uses `0.18` opacity rather than full hide because a complete hide breaks spatial context — operators want to see *where* the focused subgraph sits in the bigger network, not just the subgraph itself; `0.18` is dim enough that the spotlight reads instantly but visible enough to keep peripheral context.
**Alternatives considered:** Speeding the visible expansion via higher `alphaDecay` instead of pre-warming (rejected — still leaves a visible expand cycle that has no narrative benefit; pre-warming makes the first paint correct rather than racing to converge). Running the warmup in a `useLayoutEffect` instead of inside useMemo (rejected — useMemo runs in render so the converged positions are visible to the very first JSX read of `positionsRef`; useLayoutEffect would require an extra render cycle and a placeholder). Cluster-collapse for dense subgraphs as the visibility fix (offered to user, deferred — solves a different problem (visual density at low zoom for dense regions) and would be a separate feature; warmup + bigger nodes addresses the immediate "I can't see them" complaint). Solid (non-translucent) node fills (rejected for this pass — would lose overlap legibility for dense clusters; bigger radius + adaptive scale was a less invasive lever). Marker fill via hardcoded color for highlight (rejected — `fill="context-stroke"` is exactly the right primitive and removes a class of color-sync bugs). Per-layer arrow markers (rejected — only structural edges have a meaningful direction; staker/owner/reward are symmetric clique edges where an arrow would mislead). Hiding non-relevant nodes entirely on selection (rejected — loses the "where in the network" context that the user values).

## Decision #71 - 2026-05-08
**Context:** Post-#70 polish on the network graph branch surfaced four interaction/visual issues during real-world use of `/network`: (a) the selected-node primary-500 pulsing ring + static ring read as out-of-place when the selected node was a CRN (the green node got a purple ring); (b) plain-`<span>` node labels were hard to scan and needed visual weight per kind; (c) overlay layers (`owner`, `staker`, `reward`) crushed their members into dense knots while structural CCN→CRN pairs floated loosely; (d) initial fit / reset-view felt too zoomed-in — bumping the padding factor alone produced no visible change.
**Decision:** (1) Replace the selected-node accent (pulsing primary-500 ring + static primary-500 ring) with a single filled translucent halo at `r + 8` in the node's own `color` at `fillOpacity=0.25`, rendered behind the node body. No animation. The reward-kind branch uses a rect halo for shape parity. (2) Render CCN/CRN labels as DS `Badge` (`fill="outline"`, `size="sm"`) with a kind/status-mapped variant via a `labelVariant()` helper: `inactive → info`, `unreachable → error`, `ccn → default` (purple), `crn → success` (green). (3) Drop the constant `forceLink.strength(0.5)` override; let d3-force's degree-aware default (`1 / min(degree(source), degree(target))`) apply. (4) Loosen the initial fit: bounding-box padding `1.4 → 2`, upper zoom cap `4 → 2`, and `MIN_FIT_ZOOM` floor `0.6 → 0.3`.
**Rationale:** **Same-color halo** preserves the kind signal (purple stays purple, green stays green) instead of imposing a primary-500 accent on every selection. The pulse animation is now exclusive to the address-deep-link highlight, so when a node is both `?address=`-matched and clicked, the two states remain visually distinguishable. **Badge labels** carry the kind color "for free" via DS variants, removing the need for a separate label-coloring pass and giving each name its own visual chip — much easier to scan in dense areas. **Degree-aware link strength** is the d3 default for a reason: with a constant strength, cumulative pull on a clique member scales linearly with degree, so a 6-member same-owner clique pulls each member with `(N−1) × strength` — several times harder than a sparsely-linked structural pair. The default normalizes this. **Loosening the fit** required all three changes at once: just bumping the padding had no visible effect because `MIN_FIT_ZOOM = 0.6` was clamping zoom regardless. With the floor at `0.3` and the cap at `2`, the new padding factor takes effect for the full graph and the focus/address ego subgraphs both fit with breathing room. Validated by user: "feels way better."
**Alternatives considered:** Per-layer link strength (`structural` snappy, overlays loose) — rejected; degree-aware default already handles the asymmetry without per-layer tunables. Keeping the pulse animation on selection — rejected; address-deep-link already uses pulse, doubling up was visually noisy. Lowering `MIN_FIT_ZOOM` to `0` — rejected; outliers in the simulation could make the graph too tiny; `0.3` keeps a sane minimum. Stripping the trailing `*` some operators put in their CCN names — rejected (Decision noted in conversation); the asterisks come from the corechannel registry and stripping them in one display surface but not others would be inconsistent.

## Decision #70 - 2026-05-08

## Decision #20 - 2026-03-04
**Context:** api2.aleph.im hash lookups fail with 400 when sending all 426 VM hashes in a single GET request (~28KB URL)
**Decision:** Batch hash lookups into chunks of 100, fetch in parallel with `Promise.all`
**Rationale:** The api2 endpoint only supports GET (POST returns 405). Browser/server URL limits are ~8KB. 100 hashes × 66 chars ≈ 6.6KB per request, safely under the limit. `Promise.all` keeps latency low — 5 concurrent requests complete in roughly the same time as one.
**Alternatives considered:** POST request (not supported by api2), reducing the number of VMs queried (would miss the truly latest ones), server-side proxy (adds infrastructure)

## Decision #19 - 2026-03-04
**Context:** Adding Latest VMs card to overview page — VM creation timestamps not available from the scheduler API
**Decision:** Progressive loading from two APIs — scheduler data renders immediately, creation timestamps from api2.aleph.im arrive asynchronously via a separate React Query hook
**Rationale:** The scheduler API has no `createdAt` field. VM hashes are Aleph message `item_hash` values, so `api2.aleph.im/api/v0/messages.json?hashes=...` returns the creation timestamps. Rather than blocking the card render on both APIs, we show scheduler data immediately (hash + status badge) with inline Skeleton placeholders for timestamps. Once api2 responds, rows re-sort by creation time. `staleTime: 5min` and no polling since creation timestamps are immutable.
**Alternatives considered:** Single combined API call (scheduler doesn't have the data), blocking render until both APIs respond (worse UX — scheduler is fast, api2 is slower), storing creation times in local cache permanently (timestamps never change but cache invalidation is simpler with React Query's staleTime)

## Decision #18 - 2026-03-04
**Context:** `hasVms` checkbox on nodes page caused a visible delay when toggling
**Decision:** Keep client-side filters out of the React Query key; apply them post-fetch in the component; wrap state setters in `useTransition`
**Rationale:** Including `hasVms` in the query key (`["nodes", {hasVms: true}]`) made React Query treat it as a different query on toggle — triggering a new network request + loading state for what's actually just a client-side `vmCount > 0` filter. Moving filtering into the component makes the toggle instant (same cached data, different view). `useTransition` defers the expensive table re-render so the checkbox updates in the current frame.
**Alternatives considered:** `useDeferredValue` on the filtered array (less explicit), keeping in query key with `keepPreviousData` (still causes unnecessary refetch)

## Decision #14 - 2026-03-04
**Context:** Migrating from mock-only data to real scheduler API integration (`/api/v0` at port 8081)
**Decision:** Full type rewrite (not incremental field renames), remove EventFeed and sparklines, derive per-status counts client-side
**Rationale:** The API's data model differs significantly from the mock model: snake_case wire format, flat resources instead of nested `ResourceSnapshot`, single VM status instead of dual scheduled/observed, `HistoryRow` instead of `SchedulerEvent`, and no global events or stats history endpoints. Incremental patching would have been messier than a clean rewrite. EventFeed removed because there's no global events endpoint — history is per-resource and shown in detail panels. Sparklines removed because there's no `/stats/history` endpoint; client-side accumulation deferred to backlog. `getOverviewStats()` fetches `/stats` + `/vms` + `/nodes` in parallel to compute per-status breakdowns not available from `/stats` alone — React Query deduplicates the concurrent `/vms` call.
**Alternatives considered:** Keeping sparklines via client-side accumulation in React Query cache (adds complexity for a feature that should come from the backend), adapter layer to preserve old type shapes (more code to maintain than a clean rewrite), keeping EventFeed with per-resource history aggregation (API doesn't support it efficiently).

## Decision #12 - 2026-03-03
**Context:** Page refresh on `/nodes` or `/vms` shows IPFS gateway directory listing instead of the app
**Decision:** Add `trailingSlash: true` to `next.config.ts`
**Rationale:** IPFS gateways do directory-based resolution. When a `nodes/` directory exists (from RSC data files), the gateway redirects `/nodes` → `/nodes/` and looks for `index.html` inside it. Without `trailingSlash`, Next.js exports `nodes.html` at the root — never found after the redirect. With `trailingSlash: true`, routes export as `nodes/index.html`, matching gateway expectations.
**Alternatives considered:** Hash-based routing (significant refactor, loses clean URLs), `_redirects` file (not supported by `aleph.sh` gateway).
